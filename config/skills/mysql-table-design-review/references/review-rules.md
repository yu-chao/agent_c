## R01 字典码字段

字典驱动字段的注释必须标注字典码，使开发者能定位字典表中的完整枚举名称。

推荐：

```sql
order_status varchar(2) COMMENT '订单状态(NEW_ORDER_STATUS)'
```

若已知字段由字典驱动但只在注释中复制枚举值，记录 ERROR。若无法确认是否使用字典，记录 MANUAL_REVIEW，并请求字典字段清单或应用配置。

## R02 废弃字段

已废弃字段的注释必须明确包含“废弃”，防止继续参与开发逻辑。

```sql
delivery_status varchar(2) COMMENT '发货状态(废弃)'
```

DDL 只能确认是否已标注，不能判断字段实际上是否废弃。没有废弃清单、代码引用或变更记录时，记录 MANUAL_REVIEW，不得臆测。

## R03 逻辑删除与可追溯性

业务数据尽量避免物理删除，以免订单商品行等历史记录失去追溯关系。问题单建议每张业务表包含：

```sql
enabled bit(1) NOT NULL DEFAULT b'1' COMMENT '是否可用'
```

缺少 `enabled`、允许 NULL、缺少默认有效值或注释不清晰时记录 ERROR。若存在等价逻辑删除字段（如 `deleted`），记录 MANUAL_REVIEW，确认组织标准后再定性。

DDL 无法证明应用是否执行物理删除。标记 MANUAL_REVIEW，说明需由人工审查删除逻辑、ORM 配置、级联行为和归档策略；不要连接数据库或执行查询。

## R04 关联字段注释

关联字段注释必须精确到目标表和目标字段，避免把业务 ID 错连到另一候选键。

```sql
supplier_company_id bigint COMMENT '供应商企业ID(cust_company_supplier.company_id)'
```

对 `*_id`、`*_code` 等疑似关联字段：

- 注释包含 `目标表.目标字段` 时，仅当目标表也在输入 DDL 中，才核验目标对象存在和类型兼容；否则标记 MANUAL_REVIEW。
- 只写“供应商企业ID”等模糊说明时记录 WARNING。
- 无法判断关联对象时记录 MANUAL_REVIEW。

## R05 创建和修改时间

每张业务表必须有非空、自动赋当前时间的创建和修改时间：

```sql
create_time datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
modify_time datetime NOT NULL DEFAULT CURRENT_TIMESTAMP
  ON UPDATE CURRENT_TIMESTAMP COMMENT '修改时间'
```

检查字段存在性、NOT NULL、默认值，以及修改时间的 `ON UPDATE`。组织另有精度标准时以更严格标准为准。

DDL 只能检查字段定义，不能确认既有数据是否存在 NULL 或创建时间晚于修改时间；这些数据质量事项标记 MANUAL_REVIEW，不生成查询语句。

## R06 非字典枚举

非字典枚举字段的注释必须同时写明字段含义和完整的“值:含义”映射。

```sql
status tinyint COMMENT '询价单状态@0:已取消,1:待提交,2:已提交',
cancel_reason_type int COMMENT '取消原因@1:超时未报出,2:内容错误,3:客户取消'
```

`status`、`type`、`state`、`reason` 等字段若注释只有名称、只有数字、映射格式含糊或疑似缺值，记录 WARNING。DDL 无法证明枚举是否完整，不根据猜测升级为 ERROR。

## R07 字段注释

所有字段必须有注释，且注释应与业务名称一致。缺失或空注释记录 ERROR；“字段1”“备注”等无法解释用途的注释记录 WARNING。业务一致性无法仅靠 DDL 证明时记录 MANUAL_REVIEW。

## R08 表名、表注释与业务一致性

表名、表注释和真实业务概念必须一致。问题单案例中，表名 `oms_third_purchase_order` 应对应“第三方采购订单”，而不是“预订单”。

有明确业务定义且名称或注释冲突时记录 ERROR；没有业务定义时只记录 MANUAL_REVIEW，不根据英文拆词自行判定。

## R09 字典与业务枚举一致性

同一枚举不得长期由业务表、代码常量和字典表维护互相冲突的两套标准。核对：

- 业务数据出现的值是否都存在于字典；
- 字典是否缺少当前代码或业务使用的选项；
- 显示名称、启用状态和排序是否一致；
- 谁是唯一事实来源，变更是否同步。

仅有 DDL 时始终记录 MANUAL_REVIEW。报告应说明需人工对比字典与业务值集合，但不要查询数据库，也不要要求数据库连接。

## R10 主表字段膨胀

禁止随着业务发展在主表无序堆积字段。问题单以拥有 144 个字段的 `pay_order` 为风险案例，但未规定固定阈值，因此不得把 144 或其他数字当作通用硬限制。

字段明显较多时记录 WARNING，并按业务子域分组检查：

- 是否存在已废弃字段；
- 是否混入低频扩展、流程快照、重复展示值或一对多数据；
- 是否应拆为扩展表、明细表或历史表；
- 拆表是否会破坏事务一致性和核心查询性能。

最终拆分方案必须结合查询模式、写入事务、数据规模和生命周期人工确认；仅凭 DDL 不直接判定具体拆表方案。

## DDL-only 判定边界

- 只以用户提供的 DDL 原文作为确定性证据。
- 不连接数据库，不读取 `information_schema`，不执行或建议执行 SQL 查询。
- 数据是否被物理删除、审计时间是否存在异常值、字典与业务数据是否一致，均标记 MANUAL_REVIEW。
- 目标表未包含在输入 DDL 中时，不确认关联字段是否真实存在或类型兼容。
- 用户提供多张表的 DDL 时，可以在这些 DDL 之间核对关联对象、字段类型、命名和注释。
- 用户只提供部分 DDL 时，明确评审范围，不把未提供的信息视为不存在。

报告中的人工确认项只描述需要核验的事实和所需材料，例如字典码说明、业务定义、废弃字段清单或相关表 DDL；不要提供查询模板。
